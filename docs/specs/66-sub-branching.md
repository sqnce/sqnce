# spec: engine sub-branching in @sqnce/core

Issue: #66.

Driving need: today the engine is strictly linear. A `Run` carries a single integer `frontier` over a flat `mainStages[]` list, and `advance` only ever does `frontier + 1`, either when the current stage's gate is met or when the explicit force override is used to advance past an unmet gate. There is no track / branch concept. The presales-sqnce "RFP response drafting" epic needs one pursuit that shares an upstream spine, then **forks into independent tracks that each finish at their own terminal**, with no requirement to rejoin. This issue adds that capability to the pure engine only.

## The shape, by example

```
spine:  S0 Intake  ->  S1 Findings
                          |  fork (after the last spine stage)
            track "demo"      (optional):  D0 Script -> D1 Build  -> D2 demo-QA*
            track "response"  (required):  R0 Draft  -> R1 Review -> R2 Sign-off*

(* = terminal: the track's last main stage)
mainStages[] order: [S0, S1, D0, D1, D2, R0, R1, R2]
```

- Each track advances and gates **on its own**: advancing demo never waits on response.
- **Optional** track (demo) can be marked not-applicable for a run; **required** (response) cannot.
- A run is **complete** when every required track (plus any optional track the run kept) has reached its terminal, meaning that terminal stage's boundary gate is met.
- A definition with no fork behaves **exactly** as today (one implicit track).

## Scope (confirmed during intent gate)

Engine + validation only: core logic, `validateDefinition`, JSDoc type declarations, and a new forked test fixture. **No UI**, no bundled-definition migration. One fork boundary, N parallel linear tracks, **no nesting, no rejoin, no multiple fork points**. The "clearing / cascade-invalidation" line from issue #66's body is **dropped**: this was confirmed with the owner at the issue's intent gate (recorded in the PR #72 description and this branch's commit history), so editing an upstream value never auto-wipes downstream outputs, here or anywhere. Issue #66's acceptance criteria still list a clearing item, which now disagrees with this confirmed scope; updating the issue to match is an owner action flagged for the spec-approval gate, not a silent reduction made in the spec.

## Definition schema (additive)

Two new optional fields; absence means a linear definition, validated and run exactly as today.

```js
Definition.tracks?: Array<{ id: string, name: string, optional?: boolean }>
MainStage.track?: string   // track id; absent = shared spine
```

The fork is **derived**, not a separate node: the **spine** is the untagged main stages (a non-empty contiguous prefix); the fork sits after the last spine stage; each **track** is the contiguous run of main stages tagged with its id; a track's **terminal** is its last main stage (implicit, no explicit field). `optional: true` marks a track skippable per run; absent/`false` means required. Sub-stages inherit their main stage's track; track membership lives at main-stage grain because that is the engine's only gate boundary (`frontier` is a main-stage index).

### Validation (`validateDefinition`)

When `tracks` is **absent**, the definition is linear and validated exactly as today, with one addition: a `mainStage.track` present without a `tracks` declaration is **rejected** (a stray track tag with no fork is a misconfiguration, not a silent linear run; track fields and the `tracks` declaration must both be present or both absent). When `tracks` is **present**, the following must hold, each rejection a clear problem string, covering the issue's named cases and the boundaries pinned in the intent gate:

1. `tracks` present but not an array; fewer than 2 tracks (a fork needs at least two); a track missing a non-empty `id`/`name`; duplicate track ids; a track's `optional` present and not a boolean (mirrors the existing `skippable` boolean check); a track `id` that is a reserved object-prototype key (`__proto__`, `constructor`, `prototype`). The reserved-key rejection is a conservative hardening choice, not a language guarantee: track ids become run-state map keys, and while only `__proto__` resists becoming an own property under plain assignment, `constructor` and `prototype` would become ordinary own properties (the run store deliberately supports a `constructor` run id via own-property guards), so the engine refuses all three up front rather than relying on that distinction.
2. a `mainStage.track` that is present but not a non-empty string, or that references an undeclared track.
3. an **empty spine** (stage 0 already tagged: the fork would have nothing shared before it; AC says "fork after a shared stage").
4. a **shared stage after the fork** (an untagged main stage appearing after the first tagged stage = an implicit rejoin).
5. a **non-contiguous track** (a track's stages interleaved with another's), so each track is a single linear run.
6. a declared track that **owns no main stage** (unreachable track / no terminal).
7. a declared `subject` (when present) that does not resolve to exactly one spine step whose named output exists, is a `fields` output, and declares the named field. (This is a new forked-validation rule, not a description of current runtime: `resolveSubject` blindly property-indexes whatever value is stored, `outputs[outputId][field]`, so at runtime it can read a property off a `data` object or even a `file` value (which carries `name` / `content`); only a `fields` output, though, has a declared field schema that validation can check ahead of time, so the forked rule narrows subjects to a declared `fields` output for predictable, validatable resolution. `text` and `link` values are plain strings with no addressable field.) A subject that does not resolve at all, resolves to more than one step, or resolves to a step **outside the spine** (sitting in a track) is rejected too, because a not-applicable track must never be able to strand subject resolution. Linear definitions keep today's lighter check (only `stepId` / `outputId` / `field` must be present); the stricter resolution check is added for forked definitions, where the spine placement is load-bearing.

Existing rules are unchanged (global step-id uniqueness, sub-stage id uniqueness, gate types, output specs, render/validate hints). All 8 bundled definitions and the existing linear fixture must keep validating with zero edits.

## Run state (additive)

```js
Run.trackFrontier?: { [trackId: string]: number }   // furthest committed main-stage index within each track
Run.skippedTracks?: { [trackId: string]: true }      // optional tracks marked not-applicable this run
```

Read paths use the **effective** skipped-track set, not the raw map: a `trackId` in `skippedTracks` is honored only when the definition declares that track and marks it `optional`. A required or unknown id (reachable through stale, imported, or corrupted persisted state) is ignored everywhere, so bad state can never drop required work, hide it from gates, or let a run complete without it. The requiredness guard therefore lives at every read, not only on the `skipTrack` write, because runs are JSON-persisted and a write-time guard alone is insufficient.

`validateDefinition` rejects a track id that is a reserved object-prototype key (`__proto__`, `constructor`, `prototype`), so a valid definition never writes such a key. The rejection is a hardening policy rather than a language guarantee: only `__proto__` fails to become an own property under plain assignment (`trackFrontier["__proto__"] = 0` mutates the prototype chain, not an own key), whereas `constructor` and `prototype` would become ordinary own properties, so the engine refuses all three up front instead of depending on that distinction at every read. Reads of `trackFrontier` and `skippedTracks` additionally use own-property checks (`Object.prototype.hasOwnProperty.call`, as `cloneRun` already does for store entries), never bare key truthiness, so even a corrupted persisted run cannot have an inherited property name read as a real frontier or skip entry. The broader run-id hardening of this class is tracked in #69; this spec simply does not add a new instance of the hole.

`frontier` keeps its exact meaning: the furthest committed **spine** stage. For a linear definition the spine is the whole process, so `frontier` ranges over every stage exactly as today and **neither new field ever appears** (a linear run is byte-identical to a current run). For a forked definition, `frontier` tops out at the last spine stage; `trackFrontier` appears the moment the fork opens. A stale persisted linear run can carry both a `frontier` and an `idx` that point deeper than the new spine (the definition gained a fork after the run was saved), and the engine normalizes **both** through a single fork-aware step: it clamps `frontier` to the last spine index, and when `idx` lands on a now-tracked stage that the clamped `frontier` no longer reaches, it recenters `idx` to the last committed spine sub-stage. This normalization is a single shared step that runs at the entry of every read path before any consumption of `frontier` or `idx`, and it covers two kinds of consumption. First, control-flow reads of `frontier` / `idx`: `advance`, `browse`, `jumpTo` (and `lastIndexInMain` through them), `isRunComplete`, `trackStatus`, and `skipSubStage` (whose region guard reads `frontier` / `trackFrontier`). Second, run-aware validator evaluation: the sanitized relation-set run handed to validators (described below) is built from the normalized run, so a validator can never read a stale `frontier` / `idx` either; this covers every path that evaluates validators through `gateProgress` (`advance`, `runSummary`, `isRunComplete`, `trackStatus`) and through `isStepComplete` (`buildContext`). `runSummary`'s own tally still does not read `frontier` / `idx`, but because it evaluates run-aware validators it is made safe through this second route, not by being exempt. `buildContext` additionally takes the card's flat index as an explicit argument, so deriving that index from `run.idx` is normalized by the caller that computes it. Clamping `frontier` alone would not be enough, because `advance` and `browse` anchor on `idx`: a stale `idx` sitting inside a tracked stage would otherwise make `advance` a silent no-op (its `cur.mainIndex !== frontier` guard would never match) exactly where the fork must be able to open. The run is thus normalized to "spine committed, fork openable" rather than stranded; per the pre-publish no-migration stance, that normalization is the only concession and deeper recovery of such a run is not promised.

**Representation choice.** Keep `frontier` as the spine pointer and add a per-track frontier map (chosen), versus replacing `frontier` with a single lane->index map covering spine and tracks uniformly (rejected: it changes the linear run shape, breaking "linear unchanged" and the demo seeds, which use a plain integer `frontier`). The chosen shape is purely additive and leaves the untouched UI and seeds working for every linear definition. Active / complete / skipped track status is **derived** from `trackFrontier` + gates + `skippedTracks`, never stored (single source of truth).

## Engine behavior

For a forked definition, `flattenSubStages` annotates each **tracked** flat sub-stage with its `track` (the owning main stage's track id) and that track's `optional` flag (from `definition.tracks`), so every `subStages`-based read path can compute the effective skipped-track set without separately threading the definition. A spine sub-stage carries no `track` or `optional` field, and for a linear definition no flat record gains any new field at all: where there is no track to name, the flattened output stays byte-identical to today. `advance(run, subStages, { force, validators })` reads the centered card's region from that annotation:

- **Spine, not the last spine stage:** advance the spine as today (gate = the spine stage aggregate; on success `frontier + 1`, `idx` to the first sub of the next spine stage; an advance forced past an unmet gate records `forces[frontier]`, while a met gate records nothing even when `force` is passed, exactly as today). Unchanged path, and the only path a linear definition can take.
- **Last spine stage (the fork boundary):** advancing **opens or repairs the fork**, computed per declared track rather than on map presence. On a met (or forced) gate, for every declared track that lacks a valid `trackFrontier` entry set it to that track's first main-stage index, and preserve every existing valid frontier; so a fresh open initializes all tracks, a repeated advance never resets progress (browsing back is safe), and a partially corrupted run missing a required track self-repairs instead of dead-ending. If this advance initialized at least one track (fresh open or repair), `idx` lands on the first sub-stage of the first **non-skipped** track in flat `mainStages[]` order (the track whose stage block appears earliest, which may differ from `definition.tracks` declaration order; "first" everywhere in this spec means this flat-stage order, skipping any optional track already marked not-applicable), or the last spine sub-stage when every track is skipped; if nothing needed initializing, the advance is a no-op and `idx` is unchanged. `frontier` stays at the last spine index. An open forced past an unmet boundary gate records `forces[lastSpineIndex]` as usual, while a met gate records nothing. A persisted `trackFrontier` value outside its track's stage range is treated conservatively as not-yet-at-terminal (completion stays false), never trusted.
- **Inside a track, not at its terminal:** advance that track only (gate = the current track-stage aggregate; on success `trackFrontier[t]` to the next stage in `t`, `idx` to its first sub; an advance forced past an unmet gate records `forces[currentStageIndex]`, a met gate records nothing).
- **At a track terminal, while browsing a committed stage, or centered in a skipped track:** no-op (nothing commits past a terminal, and a not-applicable track is never progressed), matching today's last-stage behavior.

`browse` / `jumpTo` / `lastIndexInMain` bound movement by the committed **reachable set** of flat indices: the spine prefix (`mainIndex <= frontier`), plus, for each open **non-skipped** track, that track's flat range up to its committed frontier (`mainIndex <= trackFrontier[t]`). For a forked definition this set is **non-contiguous**: a track committed only partway leaves a gap of unreachable flat indices (its uncommitted tail) between one open block and the next, so navigation cannot be a plain `idx + 1` step, which would land in that gap and stall. `browse` therefore interprets `direction` by its **sign** and moves to the next reachable flat index in that direction (a single reachable position), skipping any unreachable gap; the navigation callers pass `+1` / `-1`, the only magnitude that occurs. `jumpTo` accepts a target only when it is exactly a member of the reachable set (an unreachable gap index is a no-op). A track in `skippedTracks` is **excluded from the reachable set**: `browse` / `jumpTo` cannot enter it (a not-applicable branch is off the table; un-skip is by `trackId`, not by navigating in). Navigation can thus move back into the committed spine and across to another open non-skipped track even when an uncommitted tail sits between them. For a linear definition the reachable set is the single contiguous prefix `<= frontier`, so `browse` collapses to today's `idx + direction` step and `jumpTo` to today's single bound, exactly.

**Invariant:** `idx` is always inside the reachable region (the committed spine plus open non-skipped tracks). Three paths could violate it and each restores it: opening the fork lands `idx` on the first non-skipped track (or the last spine sub-stage when every track is skipped), `skipTrack` recenters `idx` out of a track it just made non-navigable, and loading a stale persisted run recenters a now-unreachable `idx` back to the last spine sub-stage (the same normalization that clamps `frontier`).

`buildContext` becomes track-scoped while preserving today's exact within-stage behavior. It still gathers every completed step in the main stages **before** the card's own main stage, plus the completed sibling steps **within** the card's own main stage (any sub-stage of it, not only those before the card), and still excludes only the drafted step and any skipped sub-stage. The single addition for a forked definition is a track filter: a card in track `t` sees the spine and `t`'s own stages but **not** any sibling track, and a card in the spine sees only the spine. For a linear definition there is no track to filter against, so the context is identical to today (the same earlier-and-current-stage set). A demo draft never sees response outputs.

For a forked definition, validators are evaluated against a **track-scoped run**, so cross-track isolation holds for gating, completion, and status, not only draft context. When the engine evaluates a step's validators it passes a sanitized run scoped to the step's **relation set** (the spine plus the step's own track, defined by topology and independent of how far either has been committed, so an intra-track validator keeps its run-aware #62-64 reach to other steps in its own track, including not-yet-committed ones; this is a distinct notion from the navigation "reachable set" above, which is bounded by the committed frontier), built as an **allowlist** so a future field cannot silently leak: `stepState`, `skips`, and `forces` filtered to that set; `trackFrontier` limited to the current track; the **normalized** spine `frontier` preserved (the clamped value, never a raw stale one); `idx` set to the validated step's sub-stage; `skippedTracks` and any other field omitted. So neither `getStepEntry` nor a direct read of `ctx.run` can reach a sibling track's outputs, progress, skip, force, or navigation state, and a validator's pass/fail or problem message can never depend on or surface sibling-track state. This threads through the **existing** `opts` without a breaking signature change: the flattened topology reaches the gate evaluation via a new optional `opts.subStages` (already on hand in `advance` / `runSummary` / `isRunComplete` / `trackStatus`); `gateProgress(subStage, run, opts)` and `mainGateProgress(mainStage, run, opts)` keep their parameter lists and scope the validator run when `opts.subStages` describes a forked topology, else pass the run through unchanged. For a linear definition the relation set is the whole run and no scoping occurs, exactly as today: the run-aware #62-64 contract is unchanged for intra-track and spine relation; only the illegitimate cross-track reach is removed. One reachability rule thus closes the non-leak guarantee for draft context and for gating, completion, and status.

The guarantee is defined over the engine's own gate-evaluating and aggregating functions, the ones that carry the topology and build the scoped run: `gateProgress` / `mainGateProgress` (when handed `opts.subStages` for a forked definition) and their callers `advance`, `runSummary`, `isRunComplete`, `trackStatus`, and `buildContext`. Each of those builds the scoped run and hands it down. `isStepComplete` stays a low-level primitive with an unchanged signature: it validates against whatever run it is given (the scoped run on the engine gate path, the full run for a linear definition). A consumer that calls `isStepComplete` directly (as the UI does for per-step display) passes its own run and is outside this guarantee, which is acceptable because cross-track validator isolation is an engine-gating property and the UI is not fed a forked definition this issue; if a consumer ever needs the isolation at that primitive, it passes a pre-scoped run rather than the engine changing the primitive's signature.

New pure helpers (all derived, none persisted):

- `isRunComplete(definition, run, opts)`: true iff (a) every non-skipped sub-stage gate along the **kept path** (the committed spine plus every track not in `skippedTracks`) is met, so a stage forced past with an unmet gate keeps the run incomplete, and (b) every kept track (required, since required tracks cannot be skipped, plus any optional track not in `skippedTracks`) has reached its terminal, meaning its `trackFrontier` equals the track's terminal stage index (for a linear definition, `frontier` at the last main stage). The reached check stops a prefilled or imported output (`stepState` is independent of `frontier` / `trackFrontier`) from reporting completion before the run advanced there; the all-gates-met check stops a forced advance past unmet intermediate or terminal work from doing so. A new but inert signal for linear definitions.
- `trackStatus(definition, run, trackId, opts)`: `"not-open" | "active" | "complete" | "skipped"`, derived for "active / complete-track tracking". It takes the same `opts` (`validators`) as `isRunComplete` / `runSummary` / `advance`, so a validator-rejected gate is seen consistently: a track whose terminal output is rejected by its validator reports `active`, never `complete`.
- `skipTrack(run, definition, trackId)` / `unskipTrack(run, definition, trackId)` / `isTrackSkipped(run, definition, trackId)`: mirror the sub-stage skip API. `isTrackSkipped` takes the definition and returns the **effective** skip state (true only when the definition declares the track `optional` and it is in `skippedTracks`; a required or unknown id reads as not-skipped), so no consumer can treat corrupted raw state as authoritative. `skipTrack` is a no-op unless the track exists and is declared `optional` (required tracks cannot be skipped, exactly as non-skippable sub-stages cannot); it sets `skippedTracks[trackId]` and never touches `stepState`. Skipping the track currently under `idx` **recenters** `idx` to the last committed spine sub-stage (always present and reachable), preserving the reachability invariant; skipping a track `idx` is not in leaves `idx` unmoved. This differs from `skipSubStage` (which never moves `idx`, because a skipped sub-stage stays navigable, while a skipped track does not). `unskipTrack` removes the entry and drops the map when empty. The read paths that **aggregate across tracks** (`runSummary`, `isRunComplete`, `buildContext`, navigation, and `advance`'s no-op-in-skipped-track check) exclude an effectively-skipped track's sub-stages, extending the existing skipped-sub-stage exclusion to also test the sub-stage's track; they have the topology to do so (`definition.tracks`, or the `flattenSubStages` optional annotation). The single-stage gate helpers `gateProgress(subStage, run, opts)` and `mainGateProgress(mainStage, run, opts)` are **unchanged**: they report one stage's gate from its (non-sub-stage-skipped) steps and do not consult `skippedTracks`, because whether a whole track participates is the aggregating caller's concern, not a single stage's. Their signatures stay stable while the corruption guard still holds everywhere track participation is actually decided. The existing **sub-stage** skip API also becomes region-aware: `skipSubStage`'s beyond-the-frontier guard accepts a sub-stage committed in its own region (within the spine `<= frontier`, or within its track's `trackFrontier`), so a skippable sub-stage inside a kept track can be marked not-applicable instead of being permanently un-skippable; `unskipSubStage` is unchanged, and tracked sub-stage skips feed gates, `runSummary`, `buildContext`, and completion exactly as spine sub-stage skips do.

`runSummary` keeps counting gate-met across every kept sub-stage; its tally logic does not read `frontier` / `idx`, and for a forked definition it additionally excludes an effectively-skipped track's sub-stages from both counts (extending the existing skipped-sub-stage exclusion). The run-aware validators it evaluates through `gateProgress` receive the sanitized relation-set run built from the normalized run, so a stale `frontier` / `idx` cannot reach them either and its totals stay stable. `resolveSubject` / `runDisplayName` are unaffected because the subject is validated into the spine. `forces` stays keyed by main-stage index (a stage belongs to exactly one track and indices stay globally unique), so the untouched UI's `wasAdvanceForced(run, sub.mainIndex)` reads keep working.

`cloneRun` fail-fast: a full clone deep-copies the whole run including `trackFrontier` / `skippedTracks` (still via `structuredClone`). `uptoStageId` truncation that resolves to a **tracked (post-fork) stage throws** a clear error (fork-aware truncation is deferred until a consumer needs it); truncation to a **spine** stage works as today and drops the track maps (they are entirely past the cut). This converts the silent index-order corruption a fork would otherwise cause into a loud boundary error.

## Backward compatibility

A definition with no `tracks` (and no `track` on any stage) takes the existing code path through every function above; the new run fields never materialize; the run shape is byte-identical to today. The regression bar is positive, not just "files unedited": the existing engine + run-store suites, the "all bundled definitions validate" test, and `npm run build -w examples/demo` all stay green unchanged. No persistence version bump, migration, or compat shim (pre-publish stance: ship breaking changes clean; here the surface is additive anyway).

## Out of scope

- Any UI / rolodex branch rendering or layout; forked definitions are driven only in code and tests this issue. The UI keeps working because no forked definition is fed to it.
- Migrating `presales.json` or any bundled definition to a fork (the genuinely-forked presales lives in presales-sqnce; `definitions/` stays validate-only).
- Active cascade-invalidation / "editing a shared stage clears downstream tracks" (dropped by owner; not built here or as a follow-up unless re-raised).
- Nested forks, multiple fork points, rejoin / merge, cross-track step dependencies; an explicit terminal field; sub-stage-granular track membership.
- Fork-aware `cloneRun` truncation (fail-fast only).
- Per-run override of a track's requiredness (the run-level lever is only "skip an optional track").
- Changing the gate-helper parameter lists. Validator scoping is threaded through the existing `opts` object (a new optional `opts.subStages`), not a new positional parameter, so `gateProgress` / `mainGateProgress` / `isStepComplete` keep their signatures and linear behavior.
- Persistence version bump, migration, or compat shim.

## Acceptance

`npm test` passes with new tests covering:

- **Validation:** rejects each malformed topology in the list above (a stray `mainStage.track` with no `tracks` declaration, a `tracks` value that is not an array, fewer than 2 tracks, a non-boolean `track.optional`, a non-string `mainStage.track`, a reserved-prototype track id such as `__proto__` or `constructor`, empty spine, shared stage after the fork, non-contiguous track, undeclared track reference, track with no stage, a subject that does not resolve to a single spine step (unresolved, duplicated, or sitting in a track)) and accepts a well-formed fork. All 8 bundled definitions and the linear fixture still validate (suite unchanged).
- **Flatten:** for a forked definition, annotates each tracked flat sub-stage with its track id and that track's optional flag, spine entries carry neither, and a linear definition's flat records are byte-identical to today (no `track` or `optional` field added anywhere).
- **Advance:** the spine advances as today; advancing past the last spine stage opens the fork with every track at its first stage and `frontier` unchanged; opening is idempotent and per-track (browsing back and advancing again preserves every present frontier and initializes any declared track missing from `trackFrontier`, so a partially corrupted run self-repairs rather than dead-ends); advancing one track moves only that track's frontier (the sibling is untouched); a track terminal is a no-op; an advance forced past an unmet gate records `forces` by stage index, while a met gate with `force` set records nothing.
- **Navigation:** `browse` / `jumpTo` are bounded per region, move between the spine and an open track, and are identical to today for the linear fixture; `browse` scans across an uncommitted track tail (a gap of unreachable indices between two open tracks) from one open track to the next rather than stalling in the gap, while `jumpTo` accepts a reachable target and is a no-op on a gap index; the `idx` reachability invariant holds across transitions: opening the fork with the first track already skipped lands `idx` on the first non-skipped track, skipping the track under `idx` recenters it to the last spine sub-stage, and `idx` is never left on a non-navigable card.
- **Context scoping:** a draft for a step in one track includes the spine and that track, and excludes the sibling track; linear context is unchanged. For a forked definition a validator on a track step sees a sanitized run (allowlist: `stepState` / `skips` / `forces` filtered to spine plus its own track, `trackFrontier` for its own track, spine `frontier`, `idx` normalized; `skippedTracks` and other fields omitted), so a validator inspecting a sibling step's output, a sibling track's `trackFrontier` / `skips` / `forces`, `skippedTracks`, or `ctx.run.idx` finds nothing it can use and cannot affect the gate, completion, or status result or its problem message; the gate-helper signatures are unchanged (scoping rides the existing `opts`), and linear definitions get the full-run view as today.
- **Optional / required:** `skipTrack` works only on an optional track and is a no-op on a required or unknown track; a skipped track is excluded from `runSummary`, context, and `isRunComplete`; `browse` / `jumpTo` cannot enter a skipped track and `advance` is a no-op while centered in one (asserted for skipping both before and after the fork opens); a required or unknown track id present in `run.skippedTracks` is ignored by every read path (gates, navigation, `runSummary`, context, `isRunComplete`), so corrupted state cannot drop required work, and `isTrackSkipped(run, definition, trackId)` returns the effective state (false for such an id); `unskipTrack` restores it; neither touches `stepState`. A skippable sub-stage inside a kept track can be marked not-applicable (region-aware `skipSubStage`) and is then excluded from its track's gate, `runSummary`, context, and completion.
- **Run-complete:** false until every kept track (required plus any unskipped optional) has reached its terminal and every non-skipped gate along the kept path (committed spine plus kept tracks) is met; true when they are; a kept (unskipped) optional track blocks completion until its own terminal is reached and gated, while a skipped optional track is excluded; a stage forced past with an unmet gate (intermediate spine or track stage, not just the terminal) keeps the run incomplete; completion stays false when a terminal's outputs are prefilled but the frontier has not reached that terminal (`stepState` present, `trackFrontier` / `frontier` still earlier); a validator-rejected terminal output keeps both `isRunComplete` and `trackStatus` incomplete; a linear run is complete when `frontier` is at the last main stage with its gate met.
- **cloneRun:** a full clone deep-copies `trackFrontier` / `skippedTracks` (distinct objects, drivable); truncating at a tracked stage throws; truncating at a spine stage works and drops the track maps.
- **Gate helpers:** `gateProgress` / `mainGateProgress` keep their signatures and report a single stage's gate (sub-stage skips honored as today), never consulting `skippedTracks`; the aggregating paths (`runSummary`, `isRunComplete`) apply effective track-skip, excluding a skipped optional track and ignoring a required or unknown id in `skippedTracks`. Given `opts.subStages` for a forked topology, `gateProgress` / `mainGateProgress` scope the validator run to the spine plus the step's own track, so a validator on a forked gate cannot read a sibling track; `isStepComplete` stays the unchanged primitive that validates against whatever run it is handed, and a direct caller passing the full run is outside the cross-track guarantee.
- **Robustness:** a reserved-prototype track id is rejected at validation; and even a corrupted persisted run carrying such a key in `trackFrontier` / `skippedTracks` is never read as opened, skipped, or complete (own-property checks).
- **Stale run:** a forked definition loaded against a persisted run whose `frontier` (and `idx`) point past the last spine index normalizes both, clamping `frontier` to the spine and recentering a now-unreachable `idx` to the last spine sub-stage, so the fork still opens (an advance from the normalized run is not a silent no-op) and navigation and completion are never stranded; the same shared normalization runs ahead of `isRunComplete`, `trackStatus`, and `skipSubStage`, and feeds the sanitized validator run, so a stale `frontier` / `idx` cannot report a run or track complete, wrongly gate a skip, or reach a run-aware validator (including the validators `runSummary` evaluates).
- **Regression:** the existing engine and run-store suites pass unmodified; `npm run types` includes the new exports; `npm run build -w examples/demo` is green.
